# Architecture

## Overview

A browser extension that adds keyboard shortcuts to control HTML media (`<video>` and `<audio>` elements) on any webpage, including media embedded in iframes or Shadow DOMs.

## Target Browsers

- Chrome
- Edge
- Firefox
- Safari (manual macOS/Xcode conversion from the Chrome build)

Uses `webextension-polyfill` to normalize `chrome.*` vs `browser.*` APIs across browsers.

## Manifest Strategy

- Manifest V3 for Chrome, Edge, and Firefox
- Manifest is generated per browser from `manifest.template.json`
- Safari wraps the Chrome Manifest V3 extension after building for Chrome first
- Frame isolation per Manifest V3 security model

## Content Script (`src/content.ts`)

- Runs on every page, injected into all frames
- Detects `<video>` and `<audio>` elements dynamically (including Shadow DOM)
- Listens for keyboard events and translates them to media control and overlay-display actions
- Communicates with iframes via `postMessage` API for cross-frame control
- Tracks "last interacted media" to intelligently select which media element to control
- Ignores input if focused element is an `<input>`, `<textarea>`, `<select>`, or is `contenteditable`

## Smart Media Selection (priority order)

1. Active/focused element (if it's a media element)
2. Currently playing media (`!paused && !ended && readyState > 2`)
3. Last interacted media (tracked via module-level variable, verified to still exist in DOM)
4. First media element on the page (fallback)

**Edge Cases:**

- Multiple playing media: first found is selected
- Iframe media: content script runs independently per frame
- Shadow DOM: recursively searched via `findMedia()`

## Settings UI

### Quick Settings Popup (`src/quick-settings-popup.html`, `quick-settings-popup.ts`)

- Stored in `chrome.storage.sync` for cross-device persistence
- Configurable: global hotkeys enabled state and per-action key bindings

### Advanced Settings Page (`src/advanced-settings-page.html`, `advanced-settings-page.ts`)

- Options stored in `chrome.storage.sync` for cross-device persistence
- Configurable: speed limits/step, seek steps, volume step, overlay display settings, debug logging

## Key Design Patterns

**Dynamic Media Detection:**

- `findMedia(root)` recursively searches DOM and Shadow DOMs
- `MutationObserver` watches for newly added video/audio/iframe elements
- Only initializes event listeners if supported media elements or iframes exist

**Cross-Frame Communication:**

- Uses `postMessage` to forward keyboard commands to iframe content scripts
- Each frame runs its own content script instance independently
- Frame isolation by design (per Manifest V3 security model)

**Overlay System:**

- Creates temporary, absolutely positioned `<div>` over media elements
- Auto-removes after brief fade-out animation
- Displays action icons (Material Symbols) + numeric values (speed, seek duration)

**State Management:**

- `lastInteractedMedia` — tracks the most recently controlled media element
- Grouped settings model:
  - `quickSettings` contains the global enabled toggle and action key bindings
  - `advancedSettings` contains playback, seek, overlay, and debugging behavior

## Tech Stack

- TypeScript
- Vite for bundling
- Vitest for unit testing
- Playwright for feature and end-to-end testing
- `webextension-polyfill` for cross-browser API normalization

## Default Settings

### Playback Speed

- **Min:** 0.25x | **Max:** 4x | **Step:** 0.25
- Clamp at boundaries — do not wrap around

### Skip Step Sizes

- **Small** (e.g. `ArrowLeft`/`ArrowRight`): ±5 seconds
- **Medium** (e.g. `J`/`L`): ±10 seconds
- **Large** (e.g. `[`/`]`): ±30 seconds

### Volume

- **Step:** 5% per keypress (0–100% range, clamped at boundaries)

## Key Technical Challenges

### Finding the Correct Media Element

Pages often contain multiple `<video>` and/or `<audio>` elements (ads, thumbnails, background videos). The content script must be injected into every frame and use `postMessage` for cross-frame communication.

### Site Conflicts

Sites like YouTube have their own native keyboard shortcuts. Needs `stopPropagation`/`preventDefault` logic and ability to disable plugin shortcuts per site.

### Safari Conversion

Do not target Safari during active development. Build and stabilize for Chrome, Edge, and Firefox first, then run the Xcode converter as a final step.

### Closed Captions (Cross-Browser)

Native `<track>` elements can be toggled via the `TextTrack` API. Platforms like YouTube use custom CC UI, so per-site handling will be required.
