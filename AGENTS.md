# Agent Instructions

Instructions for AI coding agents working on the Media Hotkeys browser extension.

For architecture details, see [ARCHITECTURE.md](ARCHITECTURE.md).
For development setup and commands, see [DEVELOPMENT.md](DEVELOPMENT.md).

## Quick Start

```sh
npm install
npm run build:chrome
npm test
npm run test:e2e    # requires build:chrome first
```

## Workflow Rules

1. **Build before E2E tests.** Always run `npm run build:chrome` before `npm run test:e2e`. E2E tests load the built extension from `dist/chrome/`.
2. **Run unit tests after any `src/` change.** Run `npm test` to verify nothing is broken.
3. **Run E2E tests after changes to `content.ts` or `overlay.ts`.** These files control behavior that unit tests alone cannot fully verify.
4. **Lint and format code after changes to `src/`, `scripts/`, or `tests/`.** Run `npm run lint` to catch style issues and `npm run format` to apply standard formatting.
5. **Do not target Safari directly.** Safari support comes from converting the Chrome build via Xcode. Never modify `dist/safari/` by hand.
6. **Keep `dist/` out of commits.** The `dist/` directory is generated by the build script and should not be checked in.

## Conventions

- **TypeScript strict mode.** All source files are TypeScript.
- **No default exports.** Use named exports everywhere.
- **Always use `webextension-polyfill`.** Never use raw `chrome.*` APIs — import from `webextension-polyfill` for cross-browser compatibility.
- **Module-level state in `content.ts` is intentional.** The content script uses module-level variables (`lastInteractedMedia`, `listenersActive`, etc.) for per-frame state. Do not refactor these into classes.
- **Manifest V3 only.** Do not add Manifest V2 fallbacks.
- **No `host_permissions`.** The extension uses `content_scripts` with `<all_urls>` match pattern. Do not add `host_permissions` to `manifest.json`.

## Common Tasks

### Add a new hotkey action

1. Add the action name to the `MediaAction` type in `src/storage.ts`
2. Add a default key binding in `src/settings/defaults.json`
3. Implement the handler in `src/content.ts` (in the action dispatch logic)
4. Add overlay icon/label support in `src/overlay.ts` and `src/icons.ts` if visual feedback is needed
5. Add a label in `ACTION_LABELS` in `src/options.ts`
6. Add unit tests in `tests/content.test.ts`
7. Add E2E test coverage in `tests/e2e/extension.spec.ts`

### Add a new configurable setting

1. Add the setting with its default value in `src/settings/defaults.json`
2. Add the type and storage key in `src/storage.ts`
3. Add UI controls in `src/options.html` and `src/options.ts`
4. Wire it into the relevant behavior in `src/content.ts` or `src/overlay.ts`
5. Add tests in `tests/options.test.ts`

### Build and package for release

1. `npm run build` — builds for all browsers
2. `npm run package` — creates distributable archives in `dist/packages/`
3. Verify Chrome and Firefox packages manually before publishing

## Testing

### Unit Tests (Vitest)
- Tests are in `tests/*.test.ts`
- Environment: `jsdom`
- Mock browser APIs with `vi.mock`
- Run: `npm test` or `npm run test:watch`

### E2E Tests (Playwright)
- Tests are in `tests/e2e/`
- Loads the built Chrome extension into a real Chromium instance
- Simulates keypresses and asserts on media element state
- Run: `npm run build:chrome && npm run test:e2e`
- Single worker, no parallelization (extension context requires it)

## Gotchas

- **Frame isolation:** The content script runs independently in every frame. Changes to cross-frame behavior must account for `postMessage` coordination between parent and child frames.
- **MutationObserver:** Media elements can appear dynamically (SPAs, lazy loading). The `MutationObserver` in `content.ts` handles this — make sure new detection logic integrates with it.
- **Multiple media elements:** Pages may have many `<video>`/`<audio>` elements (ads, thumbnails, background). The smart selection logic in `content.ts` determines which one to control — test with multi-media pages.
- **`contenteditable` and inputs:** Hotkeys must be suppressed when the user is typing in form fields or contenteditable regions. This check is in `content.ts` — preserve it.
- **Version sync:** `manifest.json` and `package.json` versions must always match. Update both when bumping versions.
